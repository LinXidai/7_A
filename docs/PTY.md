## 交互式终端实现：基于 PTY 的伪终端桥接（核心增强）
仅依赖 `PIPE` 虽然可以实现普通流式输出，但对于 `input()`、Python REPL、shell 提示符等 **交互式程序** 支持并不理想。其根本原因在于：  
很多交互式程序会根据“自己连接的是终端（TTY）还是普通管道（PIPE）”来决定输出与刷新策略。例如，`input("请输入: ")` 输出的提示词通常**没有换行符**，若使用基于 `readline()` 的管道读取，该提示词往往要等到后续真正出现换行时才会被上层捕获，导致界面上看起来“卡了一下才显示”。

为了解决这一问题，本模块在 POSIX/Linux 环境中引入了 **PTY（Pseudo Terminal，伪终端）模式**，实现机制如下：

- 通过 `pty.openpty()` 创建一对 master/slave 文件描述符；
- 将子进程的 `stdin`、`stdout`、`stderr` 全部绑定到 **slave 端**；
- 父进程（即 TUI）持有 **master 端**，并像读取真实终端一样异步监听其输出；
- 用户在输入框中提交的内容，不再写入 `process.stdin` 管道，而是通过 `os.write(master_fd, ...)` 直接写入 PTY；
- 这样，子进程会把自己视为运行在一个真实终端内，因此：
  - `input()` 的提示词可立即刷新显示；
  - REPL / shell 类程序能够正常等待用户输入；
  - 输出行为更接近真实命令行终端。

此外，为避免“用户输入在 UI 中回显一次，子进程终端再回显一次”造成重复显示，系统会关闭 PTY 从端的 `ECHO` 标志，即由 UI 自己负责展示用户输入，而不让终端驱动重复回显。

在输出监听方面，本模块未再依赖 `readline()`，而是直接对 PTY master 进行事件驱动式读取（配合事件循环的 reader 回调）。这意味着即使某段输出**没有换行符**，只要子进程已经写出字节流，前端也能第一时间获取并显示。  
这正是交互式程序提示词能够“立即出现”的根本原因。

> 补充说明：当前 PTY 方案主要适用于 POSIX/Linux 环境；在不支持 PTY 的平台上，可退回到 `PIPE` 模式作为兼容方案，但交互体验会弱于 PTY。
